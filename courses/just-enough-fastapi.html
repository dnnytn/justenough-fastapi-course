FastAPI sits between clients and your data/services — it receives HTTP requests and returns HTTP responses

FastAPI is a Python web framework. Its one job: receive HTTP requests, run your handler functions, and return HTTP responses. It is not a database, not a reverse proxy, not a queue.

It runs as a long-lived process (via uvicorn, an ASGI server) that listens on a port — typically 8000. The reverse proxy in front routes traffic to it. FastAPI never touches the network layer below the application level.

The framework ships with two things that make it distinctive: automatic OpenAPI docs (browse /docs on any running FastAPI app) and Pydantic-powered request/response validation built into the route declarations. Both fall out of the type annotations you write anyway.

Use FastAPI for: REST APIs, JSON microservices, webhook receivers, AI inference wrappers.
Not for: serving HTML pages (use templates + Jinja2, or just Django), stateful websocket servers at scale, or batch processing jobs — run those as separate processes.

§2

How a Request Becomes a Response

1
HTTP Request arrives
uvicorn accepts the TCP connection and parses the raw HTTP into an ASGI scope. FastAPI receives it as a Request object.
2
Middleware stack (inbound)
Each middleware sees the request before routing. CORS headers are checked here. Auth can be enforced here. Logging, rate limiting, tracing — all middleware.
3
Route matching
FastAPI compares the method + path against registered routes. First match wins. No match → 404. If a route matches but the method doesn't → 405.
4
Dependency resolution
FastAPI resolves Depends() callables in topological order before calling your handler. A failed dependency can short-circuit with an HTTP error.
5
Pydantic validation (request)
Path params, query params, and the request body are coerced and validated against your declared types. Invalid input → automatic 422 with a structured error body. No validation code needed.
6
Your handler runs
Your function receives clean, typed Python objects. It does its work — queries a DB, calls an API, runs a calculation — and returns a value.
7
Pydantic serialization (response)
The return value is validated against the declared response_model and serialized to JSON. Fields not in the model are stripped. Wrong type → 500.
8
Middleware stack (outbound)
Middleware sees the response on the way back. CORS adds Access-Control-Allow-Origin headers here. Logging records latency and status code.
9
HTTP Response returned
uvicorn writes the response bytes to the TCP socket. Done.

Every HTTP request flows through this pipeline — understanding it lets you reason about where to add behavior

The key insight: steps 5 and 7 are automatic. You declare types; FastAPI validates. You return a value; FastAPI serializes. All the boilerplate that exists in Flask or Express is gone.

Everything else in the framework — dependencies, middleware, routers — is just a way to hook into specific points of this pipeline.

§3

Routes & Methods

Task Tracker API — complete route surface
GET /tasks → list tasks (optional ?status=open&limit=50)
POST /tasks → create a task (body: JSON)
GET /tasks/{task_id} → get one task
PUT /tasks/{task_id} → replace task (full body required)
PATCH /tasks/{task_id} → update task fields (partial body ok)
DELETE /tasks/{task_id} → delete task (204 No Content)
path param {task_id} is extracted from the URL and passed to your function as a typed argument
query params ?status=open&limit=50 are declared as function params with defaults

Routes are method + path pairs. FastAPI matches them in declaration order.

A route in FastAPI is a Python function decorated with a method decorator: @app.get(), @app.post(), etc. The decorator registers the path pattern, the function handles the request.

Path parameters are denoted with curly braces in the path string (/tasks/{task_id}) and appear as typed function arguments. FastAPI extracts and coerces them automatically.

Query parameters are function arguments with no matching path segment — FastAPI looks for them in the URL's query string (?key=value). Give them defaults to make them optional.

The HTTP method (GET, POST, PUT, PATCH, DELETE) is not decoration — it's semantics the entire rest of the web understands. GET is safe and idempotent. POST creates. PUT replaces. PATCH updates fields. DELETE removes. Following these conventions is what makes your API predictable to any HTTP client.

Route order matters: FastAPI evaluates routes in the order they're declared. Put /tasks/stats before /tasks/{task_id} or "stats" will be treated as a task_id.

Pydantic models are the contract between your API and the outside world — define them first, code follows

Pydantic models are Python classes that declare fields with types. FastAPI uses them in two directions: to parse and validate incoming JSON, and to shape and filter outgoing JSON.

The practice is: define separate models for input vs output. Your TaskCreate model has the fields a client sends when creating a task. Your Task model (the response) includes the id and created_at the server generates. The response model is what clients see — everything else is internal.

You typically have three model shapes per resource: a Create model (required fields, no id), an Update model (all fields optional, for PATCH), and a Read model (the full response, includes server-generated fields).

Security consequence: if you declare a response_model, FastAPI will only serialize fields that appear in that model. Return your DB row directly — sensitive columns won't leak if they're not in the response model.

FastAPI generates 404 and 422 automatically. You generate domain errors via HTTPException. Everything else is a 500.

The 422 is automatic — you never write validation error code. FastAPI catches the Pydantic ValidationError and formats it into a structured JSON body that tells the client exactly which fields failed and why.

For domain errors (task not found, permission denied, conflict), raise HTTPException(status_code=404, detail="Task not found") anywhere in your handler or dependency. FastAPI catches it and returns the right status code.

For global error shapes (e.g., wrapping all errors in {"error": ...}), register an exception handler on the app. This runs for any exception type you specify — including custom exception classes.

Don't return errors as 200: use proper HTTP status codes. Clients, proxies, load balancers, and monitoring systems all understand 4xx/5xx semantics. Wrapping errors in a 200 breaks all of them.

Dependency Injection

FastAPI resolves and injects these before calling your handler
get_db()
Opens a DB session, yields it, closes after request
get_current_user()
Reads Authorization header, validates JWT, returns User
get_settings()
Reads env vars once, cached for app lifetime
rate_limiter()
Checks Redis counter, raises 429 if over limit
→ → → →
GET /tasks/{task_id}
handler receives db, user, settings, already resolved

Dependencies form a graph. FastAPI resolves it, handles lifetimes, and caches within a request.

Dependency injection is how FastAPI handles cross-cutting concerns without tangling them into your business logic. A Depends() declaration in a function signature tells FastAPI: "call this other function first and pass me its return value."

Dependencies can depend on other dependencies. FastAPI builds the full graph and resolves it in the right order. If any dependency raises an HTTPException, the chain short-circuits and your handler never runs.

Lifespan control: dependencies that yield instead of return get cleanup code that runs after the response is sent — perfect for DB sessions that need to be closed, or file handles that need releasing.

Dependencies declared on a router or the app itself apply to every route under them — you don't repeat the auth check on every endpoint.

Practical pattern: put all shared infrastructure (DB session, current user, settings, rate limiter) in a deps.py file. Routes import and declare them via Depends(). Business logic never touches the raw request object.

Middleware wraps the core in layers. Each layer can modify request/response or short-circuit. Declaration order = execution order.

Middleware is registered on the app with app.add_middleware(). The first middleware added is the outermost layer — it sees requests first and responses last. Declaration order matters.

CORS is the most commonly needed middleware. Without it, browsers block cross-origin API calls at the network layer. CORSMiddleware handles preflight OPTIONS requests automatically and adds the required headers to every response. You configure which origins, methods, and headers are allowed.

Custom middleware is just a class with an __init__ that takes the app and a dispatch (or __call__) method. It calls await call_next(request) to pass the request down the chain and get the response back.

CORS is a browser constraint, not a server security mechanism. Setting allow_origins=["*"] does not make your API public — it just tells browsers they're allowed to call it. Server-side auth (tokens, API keys) is what actually restricts access.

Project Structure

task-tracker/
app/
main.py ← app = FastAPI(), mount routers, add middleware
deps.py ← get_db(), get_current_user(), get_settings()
models.py ← Pydantic schemas: TaskCreate, Task, TaskUpdate
database.py ← engine, SessionLocal, Base
routers/
tasks.py ← APIRouter with /tasks routes
users.py ← APIRouter with /users routes
services/
task_service.py ← business logic (no HTTP awareness)
pyproject.toml ← dependencies
.env ← DB_URL, SECRET_KEY, etc
Dockerfile

Scale horizontally: add routers, not complexity in main.py

main.py is thin — it creates the FastAPI() app instance, registers routers with app.include_router(), adds middleware, and possibly defines the lifespan context (startup/shutdown). No route handlers live here.

Routers (APIRouter) group related routes. A router is like a mini-app that gets mounted at a prefix. include_router(tasks_router, prefix="/tasks", tags=["tasks"]) and all the routes in that router inherit the prefix. One router per resource is the standard split.

models.py holds your Pydantic schemas. If the project grows, split into models/tasks.py, models/users.py, etc. Keep DB models (SQLAlchemy/SQLModel) separate from API schemas.

Services contain business logic that has no knowledge of HTTP. A service function takes domain objects (not Request objects) and returns domain objects. This makes logic testable without starting the server.

The rule: if a function imports from fastapi, it belongs in a router or dependency. If it only imports from your models and DB layer, it belongs in a service. Keep the layers clean.
